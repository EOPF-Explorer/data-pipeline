## ⚠️ The deeper open question (resolve BEFORE rebuilding T1)

The current prod flow does **NOT** look like the assumption T1 was built on:

- The trigger (`trigger_cdse.py discover`) collapses to **one product per pass** and emits each with a
  **narrow ~2-day window**. The s1processor then searches that narrow window.
- In the live run, that search returned **only 1 S1A product — no S1D**. So the "2 S1A + 2 S1D" redundancy
  documented in the old notes did **not** appear.

**Question:** does cop_dataspace actually return off-platform (S1D) products in the narrow-window prod path
at all? If it doesn't, **T1 is solving a problem that may not exist in the current flow** — and the right
move is to *not* rebuild it (or rescope the whole caching plan's P0).

**Diagnostic to answer it** (one wide-window CDSE search; also gives the real `platform` value for the fix):
- Run a throwaway pod (s1tiling image + the `cdse-eodag-credentials` secret) that does a wide-window
  (~3 weeks), wide-bbox eodag search with `platformSerialIdentifier=S1A` and prints, per product:
  `id`, `properties["platform"]`, `properties.get("platformSerialIdentifier")`.
- Look at: (a) do any **non-S1A** products come back? (b) what is the real **`platform`** value string?
- Do this from a **standalone pod**, not by merging a configmap — see the process lesson below.

## If the premise holds — how to redo T1 correctly

1. **Match the product-id PREFIX, not the `platform` property.** Every product id starts with its platform
   code (`S1A`/`S1C`/`S1D`), which is guaranteed present and format-stable — unlike `properties["platform"]`
   whose value just bit us. Filter products whose identifier `startswith` the requested code(s).
2. Keep the rest of the structure that was fine: run for `len>=1`, pure transform `_rewrite_platform_postfilter`,
   idempotent, **raise on anchor drift** (don't silent-skip).
3. Re-add the unit tests + the plan.

## Process lesson (the important one)

**Validate the filter against real eodag results BEFORE merging the configmap.** Merging to platform-deploy
`main` auto-deploys via Flux, so "merge" = "deploy to the live cluster". The correct order is:

```
build → unit test → run the patched logic against a REAL cop_dataspace search in a throwaway pod
       → confirm it KEEPS S1A and (if applicable) DROPS S1D
       → THEN regen configmap + merge
```

The unit tests only proved the *string rewrite*; they could never catch the wrong runtime value. Only a
live search does. Don't merge the configmap on green unit tests alone.
